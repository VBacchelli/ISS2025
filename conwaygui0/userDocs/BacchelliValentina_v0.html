Introduction

Si ha a disposizione l'applicazione ConwayLife25 in Java, al momento non dotata di una GUI e con input statico fornito dalla classe ConwayInputMock.java. È richiesto ora di creare una GUI che consenta di far inserire la configurazione iniziale all'utente.

Requirements

Realizzazione di una pagina HTML che funga da dispostivo di input-ouput evoluto per l’applicazione ConwayLife25 in Java. La pagina HTML svolge il ruolo di:
  • un dispositivo di input, in quanto permette a un utente umano di settare lo stato iniziale delle celle e di inviare comandi di gestione dell’applicazione con i pulsanti start/stop/exit.
  • un dispositivo di output, in quanto permette di visualizzare lo stato della griglia.
I files relativi alla sua realizzazione sono:
  • guipage.html: pagina HTML
  • WSConwayguiLifeMqtt: gestione delle WebSocket lato server
  • iomap.js: mappa con le celle
  • commsocket.js: gestione della WebSocket lato client

Requirement analysis

Lista dei requisiti

Requisiti funzionali
  1. L'applicazione deve inviare al server la configurazione iniziale alla fine dello stadio di setting
  2. L'applicazione deve segnalare al server gli input dell'utente per richiedere o fermare l'invio delle configurazioni calcolate durante lo stadio evolutivo
  3. L'applicazione deve permettere all'utente di inserire la configurazione iniziale della griglia durante lo stadio di "setting"
  4. L'applicazione deve permettere all'utente di visualizzare le configurazioni della griglia ricevute durante lo stadio "evolutivo"
  5. L'applicazione deve contenere un pulsante "start" che permette di iniziare o continuare lo stadio evolutivo
  6. L'applicazione deve contenere un pulsante "stop" che permette di fermare lo stadio evolutivo per osservare la configurazione attuale
  7. L'applicazione deve contenere un pulsante "exit" che permette di tornare allo stadio di setting
  8. L'applicazione deve segnalare all'utente l'evento di game over, cioè la configurazione in cui tutte le celle sono morte, e poi fermare l'esecuzione
Requisiti non funzionali
  1. L'applicazione, durante lo stadio evolutivo, deve visualizzare ogni configurazione della griglia ricevuta appena questa viene scaricata
  2. L'applicazione deve gestire la comunicazione col server tramite WebSocket
  3. Il pulsante "start" non può essere premuto se lo stadio evolutivo è in esecuzione
  4. Il pulsante "stop" non può essere premuto se lo stadio evolutivo non è in esecuzione
  5. Il pulsant "exit" non può essere premuto quando si è nello stadio di setting
  6. L'applicazione deve presentare un'interfaccia esteticamente piacevole

Glossario

Term Definition
Cella Singola cella della griglia che compone l'interfaccia principale di gioco
Cella viva Cella della griglia nello stato "viva", rappresentata da un colore verde
Cella morta Cella della griglia nello stato "morta", rappresentata da un colore grigio
Configurazione della griglia Lo stato di ogni singola cella della griglia (viva o morta)
Stadio di setting Fase di inizializzazione del gioco che consiste nell'impostazione della configurazione iniziale della griglia, ovvero quali celle sono vive e quali morte a inizio gioco
Stadio evolutivo Fase di esecuzione del gioco in cui si possono vedere via via le configurazioni che si creano nel tempo seguendo le regole

Scenari

Titolo Stadio di setting
Descrizione Questo è lo stadio primario in cui si trova l'utente dove immette la configurazione iniziale e questa viene inviata al server
Attori Utente
Precondizioni Si è appena aperta l'app o premuto il pulsante exit
Postcondizioni Ci si trova nello stadio evolutivo in esecuzione
Scenario principale
  1. L'Utente immette la configurazione iniziale della griglia
  2. L'Utente preme il tasto start quando ha concluso
  3. Viene inviata la configurazione al server
  4. Si passa allo stadio evolutivo che entra in esecuzione
Scenari alternativi
  • L'invio al server fallisce e viene segnalato all'utente
Requisiti non funzionali R2NF, R4NF, R5NF
Punti aperti Basta inviare quali celle sono in un certo stato per risparmiare banda
Titolo Stadio evolutivo in esecuzione
Descrizione i visualizzano le configurazioni ricevute dal server e si controlla che il gioco non sia finito avvisando l'utente in caso contrario
Attori Utente
Precondizioni L'ultimo tasto premuto è stato start
Postcondizioni Si sono visualizzate tutte le configurazioni ricevute dal server
Scenario principale Si ripetono in modo ciclico queste fasi
  1. Viene ricevuta una configurazione dal server
  2. Viene verificato che non tutte le celle siano morte, in questo caso si ferma la ricezione e viene visualizzato un avviso
  3. Viene visualizzata la configurazione ricevuta all'utente
Scenari alternativi
  • La ricezione della configurazione fallisce
Requisiti non funzionali R1NF, R2NF, R3NF
Punti aperti Per risparmiare banda possono essere ricevuti solo i cambiamenti dalle configurazioni precedenti, possibilità di aggiunta di un'animazione per il passaggio da una configurazione all'altra
Titolo Stadio evolutivo in pausa
Descrizione Questo è lo stadio in cui si trova l'utente durante il gioco dopo aver premuto il pulsante stop
Attori Utente
Precondizioni Lo stadio evolutivo era in esecuzione e si è premuto il pulsante stop
Postcondizioni Ci si trova nello stadio evolutivo in pausa e si può vedere la configurazione più recente della griglia
Scenario principale
  1. L'Utente preme il tasto stop durante lo stadio evolutivo
  2. Viene segnalato al server di sospendere l'esecuzione e rimane visualizzata l'ultima configurazione calcolata
  3. L'Utente può premere il tasto start per riprendere l'esecuzione oppure exit per tornare allo stadio di setting
Scenari alternativi
  • La comunicazione col server fallisce e viene segnalato all'utente
Requisiti non funzionali R2NF, R3NF, R4NF, R5NF
